# Speaker notes — *How to Build Logic When You Code*

Open [`index.html`](index.html) in a browser. `→`/Space advance, `←` back, `F` fullscreen.
~30–35 minutes, 21 slides (incl. 4 fully worked real-world examples + a pattern gallery). The deck
teaches the **general method** for building logic; the 5 problems in [`README.md`](README.md) are the
practice that follows.

| # | Slide | Talk track (≈ time) |
|--:|-------|---------------------|
| 1 | Cover | "Today isn't about new syntax — it's about the thing that actually makes coding hard: figuring out the steps. Good news: it's a method, not a talent." (1m) |
| 2 | The gap | Normalise the struggle: knowing syntax ≠ knowing how to solve. Ask the room: "ever understood every line but still couldn't start?" Hands go up. (2m) |
| 3 | Definition | Split **thinking** (design the steps) from **typing** (write Python). Hammer "the computer is fast but has zero common sense — it does exactly what you say." (2m) |
| 4 | 5-step method | The spine of the talk. Point out **4 of 5 steps happen away from the keyboard**. (2m) |
| 5 | Step 1 Understand | Input / Output / Rules. Demo: have them restate a simple problem in one sentence. (3m) |
| 6 | Step 2 Example by hand | The "magic question" — what did my brain just do? Walk a tiny example live on a whiteboard. (3m) |
| 7 | Step 3 Plan / pseudocode | Show the plain-English plan. Stress: language-independent, and it becomes your code comments. (3m) |
| 8 | Pattern toolbox | The six reusable shapes. Tie each back to Days 1–5 (accumulator/counter = loops, flag = booleans, decompose = functions). (3m) |
| 9 | Step 4 Code | Plan → Python, line for line. The point: once the plan exists, coding is translation. (2m) |
| 10 | Step 5 Test | Normal / boundary / weird. "A failing test is a free bug report." (3m) |
| 11 | Divider: *See it in action* | "Enough theory — let's run the method on four everyday problems." Set up the plan→code layout. (0.5m) |
| 12 | Worked ex 1 · café bill | Conditionals. Do the ₹600 case by hand first, *then* reveal the code. Point at the boundary test (500). (2m) |
| 13 | Worked ex 2 · class topper | Loop + best-so-far. Walk the "champion" variable through 72 → 88 → 88 live. (2m) |
| 14 | Worked ex 3 · count vowels | Loop + counter. Trace "Saanvi" letter by letter. Tie back to the toolbox slide. (2m) |
| 15 | Worked ex 4 · grade calc | Decompose. Stress: two small functions (average, grade) beat one block; `grade(average(...))` composes them. (2.5m) |
| 16 | Pattern gallery | Rapid-fire: read each real task, let the room shout the pattern before you reveal it. (2m) |
| 17 | Debugging mindset | print(), rubber-duck, shrink the input, return to the plan. Most bugs = wrong logic, not syntax. (3m) |
| 18 | Pitfalls | Read the red items as confessions ("we all do these"). End on the green do-instead. (2m) |
| 19 | How to get good | De-mystify talent: it's reps and pattern recognition. Daily > binge. (2m) |
| 20 | Action | Send them to the 5 problems — rule: **plan on paper before keyboard**. (1m) |
| 21 | Close | "The plan is the program; the code just types it." (1m) |

**One thing to land:** the habit of *think first, type second*. If students start every problem by
restating it and sketching a plan, this session worked.

**If short on time:** collapse slides 6–7 (do the hand-example and pseudocode as one live demo), show
only **two** of the four worked examples (12–15), and skip slide 19. The non-negotiables are 4, 7, 8,
10, and at least one worked example (12–15).
